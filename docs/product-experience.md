# FrontierLens MVP 用户体验说明

## 核心任务

帮助 AI 从业者在 15 分钟内回答三个问题：

1. 这次模型发布是否与我有关？
2. 它真正改变了什么，为什么重要？
3. 这些判断能否回到官方原文核验？

任何不能直接帮助用户完成上述任务的功能，都不进入 MVP 主路径。

## 用户使用路径

### 第一步：判断是否值得关注（约 1 分钟）

用户首先看到模型的一句话变化、发布信号、建议阅读时间和核心变化数量。目标不是讲完技术，而是帮助用户快速决定继续阅读还是离开。

### 第二步：建立核心认知（约 10 分钟）

用户浏览四个经过筛选的核心变化。每个变化同时说明技术事实和产品影响。遇到陌生概念时，可以打开概念抽屉，通过一句话、产生原因、生活类比和工作流程建立直觉。

### 第三步：按需核对证据（约 4 分钟或更久）

用户可按照推荐阅读路线进入结构化 Tech Report，查看完整章节、页码和 FrontierLens 解读，也可以随时打开原始 PDF。Tech Report 始终是主证据，其他官方来源只作补充。

## 功能存在的必要性

| 功能 | 回答的用户问题 | MVP 是否必要 |
| --- | --- | --- |
| 发布速览 | 这次更新与我有关吗？ | 必须 |
| 核心变化卡片 | 到底改变了什么？ | 必须 |
| 概念解释抽屉 | 这个术语如何快速理解？ | 必须 |
| 产品影响翻译 | 这对我的产品或工作意味着什么？ | 必须 |
| 推荐阅读路线 | 35 页报告中我该先看哪里？ | 必须 |
| 结构化 Tech Report | 结论的上下文和原文是什么？ | 必须 |
| 原始 PDF 与页码定位 | 我能否核验证据？ | 必须 |
| 证据来源页 | 信息来自哪里，是否完整？ | 必须 |
| 官方来源监控 | 内容如何保持更新？ | 必要的后台能力，但不属于用户主路径 |

## 当前主动不做

- 搜索：只有一个模型时没有真实价值。
- 登录和头像：尚无必须依赖账户的数据。
- 阅读进度：没有可靠记录前不展示虚假百分比。
- 全局知识图谱、课程和社区：脱离发布语境的探索不能直接缩短理解一次发布的时间。
- 跨模型对比：有价值，但应在单模型理解体验验证后进入下一阶段。

## 产品设计原则

- 每个页面只回答一个主要问题。
- 先给恰到好处的理解，再允许逐层深入。
- 事实来自官方证据，FrontierLens 只生成理解。
- 不用尚未实现的按钮制造“功能很多”的错觉。

## 模型库的信息层级

侧边栏只承担“我正在看哪个模型”的定位，不承担首页信息流。层级固定为：

`厂商 / 模型家族 → 正式发布 → 模型变体`

- **模型家族**：例如 Qwen，聚合一条长期演进路线。
- **正式发布**：例如 Qwen3，是 Brief、Tech Report 与证据来源的主要归属对象。
- **模型变体**：按 Dense、MoE 等架构分组，展示 0.6B、32B、235B-A22B 等具体选型，不把八个变体拆成八条发布新闻。

后续更新遵循两条规则：

1. 新一代正式发布会成为时间线中的新节点，并保留 Qwen3 作为历史节点。
2. 同一代报告修订、权重补充或小版本更新只在当前节点显示“有更新”和变更记录，不冒充新一代模型。

## Tech Report 的认知辅助闭环

完整原文仍是第一优先级。AI 层不替换原文，只在用户遇到阅读阻力时提供四种最短路径：

1. **本节提炼**：用三个要点告诉用户这一节值得关注什么。
2. **重点标注**：在原文中标出可继续探索的关键术语。
3. **中文辅助译读**：翻译本节核心含义，并明确提示以原文为准。
4. **划词解释**：点击术语或选择文本，查看当前报告语境下的名词解释。

概念卡在此基础上回答：一句话是什么、为什么出现、如何工作、需要哪些前置知识、和哪些技术相关、如何演化、为什么不用另一种方案，以及对产品意味着什么。这样形成 `原文 → 概念 → 关系 → 直觉` 的最小闭环。

这一阶段暂不做动态图、概念实验室、完整学习路径和长期 AI Mentor。它们只有在上述闭环被真实用户验证后才进入后续版本。

## 发布语境中的技术认知地图

认知地图不是新的全局导航，也不是需要拖拽缩放的可视化玩具。它位于发布工作区中，
只回答一个问题：理解这次变化，还需要连接哪些概念？

首屏最多展示六个核心概念。点击后按“直觉 → 为什么出现 → 通俗类比 → 前置知识 →
相关技术 → 产品影响 → 官方证据”展开。规范概念跨发布复用，但每次发布的证据关系独立保存；
通用背景、系统推断与报告原文支持必须使用不同状态。没有足够证据时显示等待状态，不补写节点。

## 模型工作区与侧边栏跳转规则

侧边栏中的两个区域职责不同：

- **Qwen3 工作区导航**用于切换当前模型内的体验：发布速览、完整 Tech Report、证据与来源。
- **当前模型**只用于说明用户正在查看哪个模型家族、哪次正式发布及其变体。模型卡本身不承担页面跳转，避免和工作区导航重复。

当收录多个正式发布后，模型选择器负责切换 Workspace；切换后，三个工作区页面保持不变，只替换其内容上下文。

## 证据优先级与 Question First

来源数量和功能数量是两个维度。FrontierLens 可以支持多种官方来源，但前台仍保持一个统一的 Model Knowledge Workspace。

第一阶段证据优先级为：

1. Tech Report：1.0，方法、训练、实验和局限的主要依据。
2. Official Blog：0.8，用于补充发布背景和设计动机。
3. Benchmark：0.8，用于回答效果，并必须同时展示评测条件。
4. Official GitHub：0.7，用于回答实现、部署和开源资源。
5. Safety Report：0.8，用于回答风险、限制和能力边界。

权重是检索优先级，不是真实性评分。系统按照用户问题选择来源：问“为什么”优先报告和博客，问“怎么实现”优先 GitHub，问“效果如何”优先 Benchmark，问“安全吗”优先 Safety Report。所有 AI 生成内容都必须能追溯到原始官方证据。

## 登录后的追踪设置

用户首次登录后选择想追踪的模型家族和官方数据源。至少选择一个模型和一种来源；Tech Report 默认推荐但允许调整。设置影响发布 Feed、通知和个性化呈现范围。

官方来源由平台统一采集、去重与解析，用户偏好只控制“给谁看什么”，避免为每个用户重复抓取同一份报告。

Demo 尚未接入真实账号系统，目前使用 `demo` 身份将设置同步到后台，同时保留浏览器本地副本作为离线兜底。接入账号系统后，只需将 `demo` 替换为登录用户 ID；首次进入弹窗和侧边栏修改入口保持不变。

## 个性化发布 Feed

“我的追踪”是登录后的第一入口，只回答一个问题：我关注的 AI 模型最近发生了什么？后台先按照模型家族筛选，再按照数据源偏好筛选；同一模型最多保留最近三条，避免安全报告或模型卡数量过多淹没真正重要的发布。

Feed 条目不是新的内容孤岛。Qwen3 等已经建立工作区的发布会进入“发布速览”或“Tech Report”；尚未完成结构化的条目则先打开官方原始来源。没有匹配更新的模型继续显示“官方来源监控中”，让用户明确知道追踪仍在生效。

AI 产品经理是默认核心用户。全局 Feed 负责“最近发生了什么”，模型发布页负责“变化与产品影响”，技术原文和证据页负责“结论是否可信”。架构参数、模型变体和采集日志默认降为二级信息。

参考 AIHOT 的分层原则，全局导航、筛选列表和单条详情彼此分离；FrontierLens 不照搬其资讯分类，而将详情进一步组织为“模型家族 → 发布版本 → 文档与证据”。同一发布新增文档时并列保存并标记类型、日期和主证据；出现新版本时，在模型家族下创建新的发布节点，不覆盖历史版本。

用户可以添加自定义模型和 HTTPS 官方来源。自定义模型先进入追踪等待状态；自定义来源保存为“等待验证”，通过域名和内容审核后才会加入自动抓取白名单。
